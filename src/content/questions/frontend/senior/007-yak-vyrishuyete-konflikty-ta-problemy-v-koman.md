## Відповідь

Конфлікт у команді — це зазвичай різні пріоритети, а не «погана людина». Алгоритм: (1) 1:1 з кожною стороною — зрозуміти позицію без осуду; (2) знайти спільну ціль — продукт, дедлайн, якість; (3) факти замість думок — метрики, user feedback, cost of delay; (4) спільне рішення з явним owner і deadline; (5) зафіксувати в writing, щоб не пережовувати те саме. Якщо технічний спір (Redux vs Zustand) — time-boxed spike або POC, перемагає вимірюваний критерій. Ескалую до lead/EM лише коли блокер впливає на реліз або людина systemically toxic. Senior не «замовчує», а фасилітує.

## Приклад

Спір: «треба переписати компонент з нуля» vs «достатньо рефакторингу».

```markdown
**Факти:**
- 12 open bugs у модулі за квартал
- 40% часу спринту — hotfixes тут
- Тестів немає, coverage 0%

**Рішення:** 2-тижневий strangler — новий API компонента, старий deprecated з warning.
**Owner:** @ivan, review @maria
**Критерій успіху:** 0 P1 багів після міграції 3 consumers.
```

## Юз кейси

- Design vs dev: дизайнер хоче анімацію, що ламає LCP — пропоную спрощену версію + вимірювання
- PM тисне на scope: показую trade-off matrix — «або дата, або якість, або scope»
- Inter-team: бекенд не встигає API — узгоджуємо mock contract і parallel work
